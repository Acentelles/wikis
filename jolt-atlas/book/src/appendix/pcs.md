# Polynomial Commitment Schemes

A polynomial commitment scheme (PCS) allows a prover to commit to a polynomial, then later prove that it evaluates to a specific value at a specific point, without revealing the polynomial itself.

---

## Abstract Interface

A PCS has four operations:

1. **Setup:** $\text{Setup}(1^\lambda) \to (\text{pk}, \text{vk})$: generate public parameters (prover key and verifier key).

2. **Commit:** $\text{Commit}(\text{pk}, P) \to \text{com}$: commit to polynomial $P$. The commitment is a short string that binds the prover to $P$.

3. **Open:** $\text{Open}(\text{pk}, P, r) \to (v, \pi)$: compute $v = P(r)$ and a proof $\pi$ that $v$ is correct.

4. **Verify:** $\text{Verify}(\text{vk}, \text{com}, r, v, \pi) \to \{\text{accept}, \text{reject}\}$: check the opening proof.

---

## Security Properties

- **Binding:** A committed polynomial cannot be opened to two different values at the same point. More precisely, it is computationally hard to find $(v_1, \pi_1, v_2, \pi_2)$ with $v_1 \neq v_2$ both opening the same commitment.

- **Hiding:** (Optional) The commitment reveals nothing about $P$. KZG is not hiding by default; Pedersen-based schemes can be made hiding. Jolt Atlas does not currently require hiding commitments.

---

## KZG Commitments

KZG is the most efficient pairing-based PCS for univariate polynomials.

**Setup:** Choose a random $\tau$ and publish $[\tau^0]_1, [\tau^1]_1, \ldots, [\tau^d]_1$ and $[\tau]_2$. The trapdoor $\tau$ must be discarded ("toxic waste"); this is the trusted setup requirement.

**Commit:** $\text{com}(p) = \sum_{i=0}^{d} p_i [\tau^i]_1 = [p(\tau)]_1$

**Open:** To prove $p(z) = v$, use the polynomial remainder theorem: $p(X) - v = (X - z) \cdot q(X)$ for some quotient polynomial $q$. The proof is $\pi = [q(\tau)]_1$.

**Verify:** $e(\text{com} - [v]_1, [1]_2) = e(\pi, [\tau - z]_2)$ using the BN254 pairing.

---

## HyperKZG: Extending KZG to Multilinear Polynomials

A multilinear polynomial in $k$ variables does not fit directly into the univariate KZG framework. HyperKZG uses the **Gemini transformation** to bridge this gap.

The idea: a multilinear evaluation claim $P(r_1, \ldots, r_k) = v$ is equivalent to a chain of univariate evaluation claims connected by a specific linear relationship. HyperKZG converts the multilinear claim into $k$ univariate claims that can be batched into a single KZG opening.

Because Jolt Atlas represents polynomials in **evaluation form** (flat arrays, not coefficient arrays), no DFT/IDFT is needed. All computations are in the evaluation domain: just field multiplications.

See [HyperKZG](../how/crypto/hyperkzg.md) for the full implementation details.

---

## Trusted Setup

KZG requires a trusted setup: the SRS parameter $\tau$ must be generated by a trusted party or a multi-party computation (MPC) ceremony, and then discarded. If $\tau$ is known, a prover can open any commitment to any value.

Jolt Atlas ships pre-computed SRS files (`zkml-jolt-core/*.srs`). In production deployments, the SRS should come from a well-established trusted setup ceremony (e.g., the Hermez network's powers-of-tau ceremony).
